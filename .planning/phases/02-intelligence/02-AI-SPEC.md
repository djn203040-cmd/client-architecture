## 1. System Classification

**System Type:** Hybrid — Content Generation + Extraction

**Description:**
The intelligence layer performs two coupled AI tasks: (1) **Voice extraction** — analyzing a multi-channel corpus of the coach's real messages (Gmail, LinkedIn, WhatsApp, Instagram) to extract a structured voice profile (Layer 1) and select the 10–15 best representative examples (Layer 2); (2) **Stage-aware draft generation** — producing outbound email drafts that sound exactly like the coach wrote them, calibrated to the lead's current relationship state, informed by call transcripts, conversation history, and a continuously-updated AI lead description. "Good" means a coach reads the draft and thinks "that's exactly how I'd say it" while "Bad" means it sounds like a CRM template.

**Critical Failure Modes:**
1. **Terminal state draft leak** — a draft is generated for a lead in `unsubscribed`, `do_not_contact`, or `bounced` state. Hard block — this must never reach the queue.
2. **Voice mismatch** — the draft does not sound like the coach: wrong formality, wrong sentence rhythm, uses phrases on the never-say list, or sounds generic/AI-generated.
3. **State tone mismatch** — the emotional register is wrong for the lead state (e.g., eager/pleading tone for a no-show, or overly casual for a call-completed lead who is close to converting).
4. **Fabricated pain points** — the lead description or draft invents pain points, life context, or emotional signals not present in actual transcripts, messages, or coach notes. Creates false intimacy.
5. **Voice analysis hallucination** — Layer 1 profile contains tone adjectives, phrases, or never-say words that do not appear in the imported corpus — coach confirms a fabricated profile.

## 1b. Domain Context

> Researched and grounded in domain knowledge. Informs all evaluation rubrics.

**Industry Vertical:** High-ticket professional coaching / B2C sales follow-up

**User Population:** Independent coaching practitioners (5–10 at launch) who operate in a trust-and-brand-sensitive environment. Each coach has built a personal relationship with their leads; emails are sent AS the coach via their own Gmail account. Coaches are non-technical — they never see the AI layer, only the draft output.

**Stakes Level:** High

**Output Consequence:** Email drafts are sent under the coach's name from their Gmail account to real leads. A tone mismatch or fabricated detail can damage a real relationship, cause a lead to unsubscribe, or destroy the coach's perceived authenticity. In high-ticket sales, one bad message can close a $5,000–$50,000 opportunity permanently.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|--------------------|--------|--------|
| **Voice authenticity** | Reads as if the coach wrote it; matches their sentence length, warmth level, opener/closer style | Sounds like a CRM template; too polished; uses AI-ish phrasing ("I hope this message finds you well") | High | Coach review + never-say list |
| **State appropriateness** | Tone calibrated to relationship state — understanding for no-show, excited for call-booked, tailored-reactive for replied | Wrong register — eager/begging for a no-show; generic for a replied lead with specific objections | High | CONTEXT.md D-14 state framing guide |
| **Pain point specificity** | References actual known pain points from transcript or conversation history | Invents or assumes pain points not mentioned by the lead | Critical | CONTEXT.md D-18, D-20 |
| **Never-say compliance** | Zero prohibited phrases or words present in the draft | Any word or phrase from the coach's never-say list appears | High | Voice model Layer 1 never_say_list |
| **Contextual grounding** | Specific references to the lead's situation (career transition, deadline, stated goal) when known | Generic "I believe you can do it" with no specificity | Medium | Lead description D-18 |
| **Soft-close framing** | No-show messages are understanding then firmly finite ("last chance to rebook" is implicit, not stated as a threat) | Punishing, desperate, or passive-aggressive phrasing | High | Daniel's tone guidance in specifics |

### Known Failure Modes in This Domain

- **Generic opener contamination:** AI trained on internet text defaults to "I hope you're doing well" — the most common coach never-say. Requires explicit injection of coach's real openers.
- **Over-explaining empathy:** For no-show and call-completed states, AI tends to write multiple empathy sentences before the actual message. Real coaches keep empathy brief and move to substance faster.
- **Pain point fabrication from context gaps:** When transcript data is thin, the model fills gaps with plausible but false details (e.g., assuming a career transition because the coach works with professionals). Must detect thin context and use confidence indicator.
- **Tone drift over sequence position:** In `in_sequence` state, later-touchpoint drafts tend to become less personalized as context grows stale. Requires explicit stage position in the prompt.
- **Voice bleed across coaches:** If prompt structure ever mixes voice model data across coaches (even in testing), the coach immediately notices. coach_id scoping must be airtight at the AI call layer, not just DB layer.

### Regulatory / Compliance Context

- **CAN-SPAM / GDPR:** All outbound emails must carry a working unsubscribe mechanism (Phase 3 builds this). Phase 2 draft engine must not block or override unsubscribe state once set.
- **No fabricated testimonials or guarantees:** Drafts must not promise specific outcomes (e.g., "you'll triple your income") — this creates false advertising exposure in the coaching industry.
- **GDPR data minimization:** Lead description and voice corpus are personal data. No PII in logs or tracing data. The Anthropic API already does not train on API inputs by default — confirm at onboarding.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Coach (system user) | Primary voice-authenticity labeler — they know their own voice better than any rubric |
| Daniel (operator) | Dataset review, never-say compliance calibration, pain-point specificity spot checks |
| QA reviewer (Daniel's role at launch) | Production sampling of approved/rejected drafts to identify pattern failures |

## 2. Framework Decision

**Selected Framework:** Direct Anthropic TypeScript SDK (`@anthropic-ai/sdk`)

**Version:** `@anthropic-ai/sdk@0.39.x` (pin to latest stable at implementation time)

**Rationale:**
The Phase 2 use cases are discrete, sequential prompt calls — no agent orchestration, no retrieval pipeline, no inter-agent handoffs. The model is locked to `claude-sonnet-4-6` per CLAUDE.md. The stack is TypeScript-only. Adding LangChain, LlamaIndex, or any orchestration framework would add abstraction overhead with zero upside for this use case. Direct SDK calls are easier to debug, faster to iterate, and eliminate dependency risk. The `packages/ai-engine` package is already scaffolded with a `server-only` guard — Phase 2 fills its body.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain (TypeScript) | Abstraction overhead unjustified for discrete prompt calls; adds latency at p99; model is locked to Anthropic |
| Vercel AI SDK | Optimized for streaming UI use cases (chat interfaces); our use case is async background generation, not a chat widget |
| Claude Agent SDK | Designed for code-aware autonomous agents with tool use and MCP; overkill for parametric prompt calls |
| LlamaIndex (TypeScript) | RAG framework — no document retrieval in this system; voice examples are injected directly |

**Vendor Lock-In Accepted:** Yes — model is locked to `claude-sonnet-4-6` in CLAUDE.md. The direct SDK is the natural implementation choice. Future model changes require only a model string swap, not a framework migration.

## 3. Framework Quick Reference

> Distilled from official Anthropic TypeScript SDK docs for this specific use case.

### Installation
```bash
npm install @anthropic-ai/sdk
# Ensure ANTHROPIC_API_KEY is set in server environment only — never exposed to client
```

### Core Imports
```typescript
import Anthropic from '@anthropic-ai/sdk';
import 'server-only'; // Always — every file that touches the SDK
```

### Entry Point Pattern
```typescript
import Anthropic from '@anthropic-ai/sdk';
import 'server-only';

const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

export async function generateDraft(params: DraftGenerationParams): Promise<string> {
  const message = await anthropic.messages.create({
    model: 'claude-sonnet-4-6',
    max_tokens: 600,
    system: buildSystemPrompt(params.voiceModel),
    messages: [
      { role: 'user', content: buildUserPrompt(params) }
    ],
  });

  const block = message.content[0];
  if (block.type !== 'text') throw new Error('Unexpected content block type');
  return block.text;
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `messages.create` | Core API call — takes model, max_tokens, system, messages | Every AI call in this system |
| `system` parameter | System prompt — defines voice model, role, constraints | Inject voice model Layer 1 + Layer 2 here |
| `messages[]` | Conversation turn array — `role: 'user'` or `role: 'assistant'` | Lead context, state, transcript go here as user turn |
| `max_tokens` | Hard cap on output tokens | Set per call type: 600 for drafts, 1000 for voice analysis, 200 for lead description updates |
| `ContentBlock` | Typed response block — always check `block.type === 'text'` before accessing `.text` | Accessing generated text |
| `input_tokens` / `output_tokens` | Token counts in `message.usage` | Cost tracking + context window monitoring |

### Common Pitfalls

1. **Never instantiate `new Anthropic()` in a client component or shared utility** — the API key must stay server-side. Always import `'server-only'` at the top of the file.
2. **`message.content` is an array of blocks, not a string** — always check `content[0].type === 'text'` and access `.text`; don't assume a string response.
3. **Temperature is not a top-level parameter in the messages API** — it goes inside a `betas` or experimental path; instead, achieve voice variation through prompt variation (e.g., "write this message as you normally would — do not overthink it").
4. **Large XML payloads still count against `max_tokens` for the response** — when context is large, reduce `max_tokens` headroom is reduced only on the *response* side; input tokens are separate.
5. **The SDK does not retry by default** — add retry logic with exponential backoff for `529 Overloaded` errors using `anthropic.messages.create` in a retry wrapper.

### Recommended Project Structure
```
packages/ai-engine/src/
├── index.ts              # Public API — generateDraft, analyzeVoiceCorpus, updateLeadDescription
├── client.ts             # Single Anthropic client instance (server-only)
├── prompts/
│   ├── system.ts         # buildSystemPrompt — assembles voice model into system prompt
│   ├── draft.ts          # buildDraftUserPrompt — state-aware context assembly
│   ├── voice-analysis.ts # buildVoiceAnalysisPrompt — corpus analysis to Layer 1 + Layer 2
│   └── lead-description.ts # buildLeadDescriptionPrompt — synthesize lead context
├── context-assembler.ts  # assembleContext — orders and truncates context layers
├── token-counter.ts      # countTokens — budget calculation before API call
└── types.ts              # DraftGenerationParams, VoiceAnalysisResult, LeadDescriptionResult
```

## 4. Implementation Guidance

**Model Configuration:**
- Model: `claude-sonnet-4-6` (locked — never configurable at runtime)
- Max tokens by call type:
  - Draft generation: `600` (email drafts should be 150–400 words; 600 gives headroom)
  - Voice corpus analysis: `1500` (Layer 1 profile extraction + Layer 2 selection reasoning)
  - Lead description: `300` (concise paragraph — not an essay)
  - Lead description update (delta): `200` (append/replace specific elements)

**Core Pattern:**
Two-stage pipeline per draft generation request:
1. **Context assembly** — `assembleContext()` builds the complete prompt payload from voice model, lead description, transcript, conversation history, and state config; applies token budget allocation
2. **Generation call** — `anthropic.messages.create()` with assembled system + user prompt; result extracted and stored as a new draft row

**Tool Use:** Not applicable — this system uses no tool calls. All context is injected via prompt engineering. Tool use adds latency and complexity that is not justified for parametric generation.

**State Management:**
- Voice model state lives in `coaches.voice_model JSONB` — read once per draft generation session, not re-fetched per call
- Lead state lives in `leads.lead_status` — read at generation time; checked against hard-block list before any API call
- Draft output written to `drafts` table with `generation_context JSONB` storing the assembled prompt metadata (not the full prompt — for debugging and audit)
- Lead description cached in `leads.ai_summary TEXT` — auto-updated on new transcript or reply event

**Context Window Strategy:**
Token budget allocation for a standard draft generation call (targeting ~8,000 token input budget):

| Layer | Max Tokens | Truncation Behavior |
|-------|-----------|---------------------|
| Voice model Layer 1 (structured profile) | ~400 | Never truncate — fits easily |
| Voice model Layer 2 (10–15 examples) | ~1,500 | If over budget: drop oldest/weakest examples first (Layer 2 is ordered by quality score) |
| Lead description (AI summary) | ~300 | Never truncate — concise by design |
| State-specific instructions | ~200 | Never truncate — fixed template |
| Transcript (most recent) | ~2,000 | Truncate to first + last 500 tokens if over; use summary if available |
| Conversation history (prior messages) | ~1,500 | Drop oldest messages first; keep most recent 3 always |
| Coach notes | ~400 | Truncate to most recent note if over |
| Instruction + output budget | ~1,200 | Fixed |

## 4b. AI Systems Best Practices

> Cross-cutting patterns for building production AI systems in TypeScript.

### Structured Outputs with Zod

Use Zod for all structured AI outputs. Parse immediately after receiving the response — never pass raw AI text downstream without validation.

```typescript
import { z } from 'zod';

// Voice analysis output structure
export const VoiceProfileSchema = z.object({
  tone_adjectives: z.array(z.string()).min(3).max(8),
  formality_level: z.enum(['casual', 'conversational', 'professional', 'formal']),
  sentence_length: z.enum(['short', 'medium', 'long', 'varied']),
  emoji_usage: z.enum(['none', 'rare', 'occasional', 'frequent']),
  opener_phrases: z.array(z.string()).min(2).max(5),
  closer_phrases: z.array(z.string()).min(2).max(5),
  never_say_list: z.array(z.string()),
  selected_examples: z.array(z.string()).min(8).max(15),
});

export type TVoiceProfile = z.infer<typeof VoiceProfileSchema>;

// Parse with error recovery
async function parseVoiceProfile(raw: string): Promise<TVoiceProfile> {
  // AI returns JSON wrapped in XML tags: <voice_profile>...</voice_profile>
  const match = raw.match(/<voice_profile>([\s\S]*?)<\/voice_profile>/);
  if (!match) throw new VoiceParseError('No voice_profile XML block found in response');
  
  const parsed = JSON.parse(match[1]);
  const result = VoiceProfileSchema.safeParse(parsed);
  if (!result.success) {
    // Log validation errors for debugging, don't surface to coach
    logger.warn('Voice profile validation failed', { errors: result.error.issues });
    throw new VoiceParseError('Voice profile did not meet schema requirements');
  }
  return result.data;
}
```

### Async-First Design

All AI calls are async. Never block a request handler waiting for generation — use Inngest jobs (Phase 3) for sequence-triggered generation. For Phase 2 (manual "Generate" button), use async route handler with immediate draft row creation in `generating` status, then update to `pending_approval` on completion.

```typescript
// Route handler pattern — return immediately, update via Supabase Realtime
export async function POST(request: Request) {
  const { leadId } = await request.json();
  
  // Create draft row immediately with status = 'generating'
  const draft = await createDraftRecord(leadId, 'generating');
  
  // Fire-and-forget generation (catches internally, updates to 'error' on failure)
  generateDraftAsync(leadId, draft.id).catch(handleGenerationError);
  
  return NextResponse.json({ draftId: draft.id, status: 'generating' });
}
```

Streaming is not used for draft generation — coaches review complete drafts, not streaming text. Streaming would complicate the approval queue flow with no UX benefit.

### Prompt Engineering Discipline

**System prompt** contains:
1. Coach role definition ("You are writing on behalf of [coach name]...")
2. Voice model Layer 1 (structured profile in XML tags)
3. Voice model Layer 2 (examples in XML tags)
4. Hard constraints (never-say list, state constraints)
5. Output format instruction (write only the email body, no subject line, no preamble)

**User prompt** contains:
1. Lead state and context
2. Lead description (AI summary)
3. Transcript excerpt (if available)
4. Conversation history (most recent messages)
5. State-specific generation instruction

**XML delimiter structure** (required by CLAUDE.md):
```xml
<voice_profile>...</voice_profile>
<voice_examples>...</voice_examples>
<lead_context>...</lead_context>
<transcript>...</transcript>
<conversation_history>...</conversation_history>
<instruction>Generate a [state] message for this lead.</instruction>
```

Use explicit closing delimiters — `<voice_examples>` not `---`. Claude processes XML structure reliably for context injection.

### Context Window Management

Token budget is calculated **before** the API call using `countTokens()` (approximate via character estimation at 3.5 chars/token for English text). If the assembled context exceeds the budget:
1. Drop transcript body to summary (keep first 300 + last 200 tokens of transcript)
2. Drop older conversation history (keep most recent 3 messages always)
3. Drop all but most recent 2 coach notes
4. If still over: drop Layer 2 examples beyond the top 8 (minimum viable voice context)

The `generation_context` JSONB column on `drafts` stores which truncation steps were applied — this feeds the confidence indicator logic (D-08).

### Cost and Latency Budget

| Call Type | Est. Input Tokens | Est. Output Tokens | Est. Cost (claude-sonnet-4-6) | Target Latency |
|-----------|-------------------|-------------------|-------------------------------|----------------|
| Draft generation | ~4,000–7,000 | ~400–600 | ~$0.025–$0.045 | < 8s |
| Voice corpus analysis | ~8,000–15,000 | ~1,200–1,500 | ~$0.07–$0.13 | < 20s (background) |
| Lead description update | ~2,000–4,000 | ~200–300 | ~$0.012–$0.025 | < 5s |

**Caching strategy:** Voice model is stable between draft calls — cache the `buildSystemPrompt()` output in memory for the duration of a request (do not cache across requests; voice model updates must be reflected immediately).

**Cost control:** No rate limiting needed at launch (5–10 coaches, low draft volume). Add Upstash Redis rate limit on the generate endpoint at 20 drafts/coach/day when scaling.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric | Measurement Approach | Priority |
|-----------|--------|---------------------|----------|
| **Terminal state block** | Pass: no draft generated for `unsubscribed`, `do_not_contact`, `bounced`. Fail: any draft reaches queue for these states. | Code — check lead status before API call and after; test all 3 terminal states | Critical |
| **Never-say compliance** | Pass: zero occurrences of any word/phrase in `voice_model.never_say_list` (case-insensitive). Fail: any match. | Code — regex scan of generated draft text against never_say_list | Critical |
| **Voice authenticity** | 1–5 scale. 5: coach reads it and says "that's exactly how I'd write it." 3: recognizable but too polished. 1: clearly AI/generic. Target: ≥ 4 on reference set. | LLM judge (calibrated against coach-labeled examples) + periodic coach sampling | High |
| **State tone appropriateness** | Pass: emotional register matches state framing guide (D-14). Fail: wrong register (eager for no-show, cold for replied). | LLM judge with state-specific rubric. E.g., no-show rubric: "Does the message express understanding without being apologetic or pleading?" | High |
| **Pain point specificity** | Pass: all specific claims traceable to transcript, conversation history, or coach notes. Fail: specific claims about lead with no evidence in context. | LLM judge — give judge context + draft, ask "Does the draft make specific claims not supported by the provided context?" | Critical |
| **Lead description accuracy** | Pass: all stated pain points + context traceable to actual data (transcript/messages). Fail: fabricated details. | LLM judge + human spot-check on 10% of auto-generated descriptions | High |
| **Confidence indicator trigger** | Pass: confidence warning shown when `selected_examples.length < 8`. Fail: no warning shown. | Code — check example count against threshold | Medium |
| **Context truncation signal** | Pass: `generation_context.truncation_applied` correctly reflects which layers were dropped. Fail: truncation occurred but not recorded. | Code — verify `generation_context` JSONB on 100% of drafts | Medium |

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Terminal state block** | `lead.lead_status` is `unsubscribed`, `do_not_contact`, or `bounced` at generation time | Block — return null before any API call. Never create a draft row. Hide Generate button in UI. |
| **Never-say scan** | Every generated draft, before writing to `drafts` table | Code scan: regex match against coach's `never_say_list`. On match: flag draft with `quality_flags: ['never_say_violation']`, increment regen count, attempt one auto-regen. If second regen also fails: surface draft with prominent warning to coach. |
| **Empty voice model** | `coaches.voice_model` is null or `selected_examples` has 0 items | Block generation — return a clear error to the coach: "Set up your voice model in Settings → My Voice before generating drafts." |
| **Pain point injection from coach notes** | Coach has annotated `leads.ai_summary` (D-22) | Preserve coach edits — never overwrite coach-authored sections of `ai_summary` on auto-update. Merge new data only into non-protected fields. |
| **coach_id scope enforcement** | Every AI call payload construction | Code assertion: verify that `voiceModel.coachId === currentCoach.id` before assembling prompt. Fail loudly in dev, silently block in prod + alert. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|-----------------------|
| **Approval rate** | 100% of drafts (binary: approved vs. rejected/edited) | If weekly approval rate drops below 60%, flag for Daniel review — indicates voice model drift or prompt regression |
| **Edit distance on approved drafts** | Sample 20% of approved drafts where coach made edits | High edit distance (>30% of draft rewritten) signals voice mismatch — trigger voice model re-analysis suggestion to coach |
| **Regen rate per coach** | 100% tracking (count of regens per draft) | >2 regens per draft for a coach over 7 days → alert Daniel → investigate voice model quality or prompt template |
| **Never-say violation rate** | 100% tracking (automatic from scan) | Any violation in production → immediate investigation of never_say_list injection path |
| **Voice analysis quality** | Spot-check 5 voice profiles per month by Daniel | Catch hallucinated profile elements early; recalibrate prompts if needed |

## 7. Production Monitoring

**Tracing Tool:** Langfuse (cloud-hosted at launch; self-hostable if data residency required later)

```typescript
// Wrap every AI call with Langfuse tracing
import { Langfuse } from 'langfuse';

const langfuse = new Langfuse({
  publicKey: process.env.LANGFUSE_PUBLIC_KEY!,
  secretKey: process.env.LANGFUSE_SECRET_KEY!,
});

// Trace in generateDraft:
const trace = langfuse.trace({ name: 'generate-draft', userId: coachId, metadata: { leadId, leadState } });
const generation = trace.generation({
  name: 'draft-generation',
  model: 'claude-sonnet-4-6',
  input: { system: systemPrompt, messages },
  metadata: { truncation_applied: context.truncationSteps },
});
// After call:
generation.end({ output: draftText, usage: { input: message.usage.input_tokens, output: message.usage.output_tokens } });
```

**Key Metrics to Track:**

1. **Draft approval rate** (per coach, weekly) — primary quality signal
2. **Never-say violation rate** — compliance health; target 0%
3. **Average input token count per draft** — cost and context window health
4. **Draft generation latency p50/p95** — performance; alert if p95 > 15s
5. **Regen rate** (regens per first-draft, per coach) — voice model quality proxy

**Alert Thresholds:**

| Alert | Threshold | Channel |
|-------|-----------|---------|
| Draft generation error rate | > 5% over 1h window | Slack → Daniel |
| Terminal state block bypassed | Any occurrence | Immediate Slack alert + PagerDuty |
| Never-say violation in production | Any occurrence | Slack → Daniel + flag draft |
| Generation p95 latency | > 20s | Slack → Daniel |
| Anthropic API 529 errors | > 3 in 10 min | Slack → Daniel |

**Smart Sampling Strategy:**

Weight Langfuse traces toward human review when:
- Coach edited > 30% of the draft before approving (high edit distance signal)
- Draft was regenerated more than once
- Never-say scan flagged a near-match (token was modified post-scan)
- Draft state is `replied` (highest personalization requirement — most likely to surface voice issues)
- Lead has no transcript (thin-context path — higher hallucination risk)

Sample 5% of all other approved drafts for periodic quality baseline.
