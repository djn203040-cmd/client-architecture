## Color

Full OKLCH palette carried from Phase 2. No new colors introduced.

| Role | Light | Dark | Usage in Phase 4 |
|------|-------|------|------------------|
| Dominant (60%) — `--background` | `oklch(97% 0.008 60)` | `oklch(14% 0.010 50)` | Page surface for Settings → Notifications, Settings → Autonomous, tokenized review page |
| Secondary (30%) — `--card` | `oklch(94% 0.010 55)` | `oklch(19% 0.012 50)` | Notification matrix container, Autonomous mode card, Held tab DraftCard fills |
| Accent (10%) — `--accent` | `oklch(62% 0.14 50)` | `oklch(70% 0.14 50)` | Reserved list below |
| Destructive — `--destructive` | `oklch(55% 0.18 25)` | `oklch(65% 0.18 25)` | Mode A "Enable anyway" confirm button, Cancel-held-draft action, Disconnect channel ghost button on hover |
| Confidence amber (existing) | `oklch(72% 0.12 70)` bg / `oklch(40% 0.10 65)` text | `oklch(68% 0.12 68)` bg / `oklch(85% 0.08 65)` text | Disconnected-channel column hint, Mode A active warning banner |
| Glass surface | `bg-white/10 dark:bg-white/5 border border-white/10 shadow-[inset_0_1px_0_rgba(255,255,255,0.08)] backdrop-blur-md` | same | Celebration empty-state card, Mode A confirmation modal surface, tokenized review page DraftCard wrapper |

**Accent (`--accent`) is reserved in Phase 4 for exactly these elements — nothing else:**
1. Re-approve button on Held tab DraftCard (primary action — same accent fill as dashboard Approve)
2. "Connect" inline CTA in disconnected matrix columns
3. "Save preferences" button at bottom of Notifications tab
4. Animated checkmark stroke in celebration empty state
5. Active tab indicator on Settings sub-tabs (Notifications | Autonomous Mode | Integrations)
6. Focus rings on all interactive elements (`focus:ring-accent`) — inherited from Phase 1+2
7. "Review draft" CTA in Resend email template (HTML email button)

**Accent is NOT used for:**
- Switch component ON state (uses default neutral fill — toggle reads as state, not as primary action)
- Matrix cell "checked" indicator (uses default neutral check)
- Mode A confirmation primary action (uses `--destructive` — high friction)
- Held tab badge (uses `--muted` background — informational, not alert)
- Cancel-draft action in Held tab (uses `--muted-foreground` ghost — destructive on hover only)

**Destructive color contract additions:**
- Mode A toggle confirmation modal "Enable autonomous send" button: destructive fill (warns coach this is irreversible-feeling)
- Cancel held draft: ghost text button, hover state shows `text-destructive`. No fill — never visually loud.

**Held tab badge color contract:**
- Default: `rounded-full text-xs font-mono px-1.5 py-0.5 bg-muted text-muted-foreground`
- When count > 0: same neutral treatment — DO NOT promote held items to accent. Held is a parked state, not an alert.
- When count = 0: badge entirely hidden (per CONTEXT.md "specifics" — no empty "0" badge).

**Mode A active banner color contract:**
- Background: warm amber `oklch(72% 0.12 70)` light / `oklch(25% 0.08 65)` dark — same as confidence warning
- Text: `oklch(40% 0.10 65)` light / `oklch(85% 0.08 65)` dark
- Icon: Phosphor `WarningCircle size-3.5`
- Treatment matches Phase 2's "warm caution, not alert" pattern. Coach should feel "this is unusual" not "this is wrong."

**Slack Block Kit color note:** Slack renders against the workspace's theme. We do NOT specify hex values for Slack messages — Block Kit primary button (Approve) uses `style: "primary"` (Slack's green), danger button (Hold) uses `style: "danger"` (Slack's red), Edit uses default. Slack-rendered colors are NOT our color contract — coach's workspace settings win.

### 1. HeldTab (extension to existing DraftQueueScaffold)
**Location:** `apps/web/components/drafts/HeldTab.tsx`

### 2. Held DraftCard variant (extension to existing DraftCard.tsx)
**Location:** Edit `apps/web/components/drafts/DraftCard.tsx` — add `variant: 'pending' | 'held'` prop
**Visual delta from default `pending` variant:**
- Card border tint: `border-muted-foreground/20` instead of accent-tinted — held drafts are not "actionable now"
- Header gains a small "Held" badge: `rounded-full text-xs px-2 py-0.5 bg-muted text-muted-foreground` showing relative time mono ("Held 2h ago")
- Bottom action row replaces `Approve / Skip / Hold` with:
  - **Re-approve** — accent fill, `min-h-[44px]`, Phosphor `PaperPlaneTilt size-4 mr-2` icon. Keyboard shortcut: `R` (announced in card footer with `text-xs font-mono text-muted-foreground` shortcut badge `R`)
  - **Edit** — ghost variant, `min-h-[44px]`, Phosphor `PencilSimple size-4 mr-2` icon. Keyboard shortcut: `E`. Opens the existing `InlineDraftEditor` component (no new component) — on save, transitions back to pending status.
  - **Cancel** — ghost text-only variant, `min-h-[44px]`, no icon. Keyboard shortcut: `C`. On hover/focus: text turns `--destructive`. On click: confirm via inline two-step ("Cancel this draft?" → "Yes, cancel" / "Keep on hold") inside the card footer — NO modal. Single-line, replaces the action row briefly.
- No regen button on held variant (regen is a pending-phase capability; held drafts that need new content should be Edit-ed)

**Motion:** Re-approve triggers existing Framer exit animation (`x: 300, opacity: 0`) — card flies right like a normal Approve. Cancel triggers a softer exit (`y: 12, opacity: 0, duration: 0.2`) — quieter for a destructive-ish action.

### 3. CelebrationEmptyState
**Location:** `apps/web/components/drafts/CelebrationEmptyState.tsx`
**Elevation:** Glass card
**Structure:**
- Triggered when Pending tab count reaches zero AFTER an Approve+Next action (NOT on initial page load with zero pending — that's a different "no drafts yet" state)
- Detection: DraftQueueScaffold tracks `justEmptied: boolean` state, set true when the last pending draft transitions out via approve action
- Card structure (vertically centered in viewport, max-width 480px):
  - **Animated checkmark** at the top — SVG `path` 64×64 with `--accent` stroke, `stroke-dasharray` animated 0→100 over 600ms ease-out (Framer Motion)
  - **Display heading** (28px / 600): "You're all caught up."
  - **Body paragraph** (14px / 400 / muted-foreground): one high-signal stat formatted as: `<span class="font-mono text-foreground">{N}</span> leads responded this week` (or similar — see Copywriting Contract for full text variants)
  - **Primary CTA button**: ghost variant (NOT accent — the celebration is the moment, not the CTA), `min-h-[44px]`, text "Back to dashboard". Phosphor `ArrowLeft size-4 mr-2` icon. Routes to `/` (dashboard root).
- Card padding: `p-12` (48px all sides)
- Motion: card enters with `y: 16, opacity: 0 → y: 0, opacity: 1`, spring stiffness 120 damping 18 (matches Phase 2 VoiceProfileCard entry)
- Reduced motion: checkmark draws instantly (no animation), card enters with `opacity: 0 → 1` over 200ms

**States:** entering (checkmark drawing) | settled (card visible) | exiting (when coach clicks CTA, fade out 150ms then route)

**Structure:**

**Follow-up template** (`draft_followup_v1`, also UTILITY):

**Submission notes for Daniel:**
- Both templates submit through Twilio's content template API to Meta
- Approval typically takes 24h — submit ahead of phase ship
- Templates use UTILITY category to avoid marketing-message gating

### 12. SMSBody (short-link fallback)

## Interaction Contracts

### Approve+Next Flow (existing keyboard A key extended)

1. Coach on Pending tab, presses `A` on focused DraftCard
2. Card animates out right (`x: 300, opacity: 0`, spring damping 20)
3. Server processes approval: advisory lock → status transition → Gmail send → notification log
4. Next pending draft (by `scheduled_send_at ASC`) slides in from the bottom (`y: 12, opacity: 0 → y: 0, opacity: 1`)
5. Focus automatically lands on the new card (existing auto-focus pattern from Phase 1)
6. If no more pending drafts and `justEmptied` is true: CelebrationEmptyState renders in place of the DraftCard list

**Edge case:** If approval fails server-side (advisory lock contention, etc.), the card animates back in from the right (`x: 300 → x: 0`) and a toast appears: "Couldn't approve. Try again."

### Held Tab Interactions

1. Coach taps the "Held" tab in DraftQueueScaffold tab bar
2. Tab transitions reveal Held DraftCards in `held_at DESC` order
3. Coach can: Re-approve (`R`), Edit (`E`), Cancel (`C`) — same keyboard pattern as Pending
4. Re-approve runs the SAME approval path as Pending Approve — pre-send safety check, advisory lock, Gmail send
5. Cancel uses inline two-step confirmation (NOT modal) — replaces action row briefly with "Cancel this draft?" + "Yes, cancel" / "Keep on hold" buttons
6. Cancel commits `status = 'cancelled'`; sequence continues on schedule (next touchpoint is unaffected)

### Notification Matrix Interactions

1. Coach opens Settings → Notifications
2. Matrix loads with current preferences (default: every cell ON for connected channels)
3. Coach toggles a Switch — optimistic update, immediate server save via `POST /api/settings/notifications`
4. On save failure: Switch reverts, toast "Couldn't save preference. Try again."
5. Dashboard column: Switches are visibly ON but the toggle does not respond to clicks. Tooltip on hover/long-press: "Dashboard notifications can't be turned off."
6. Disconnected channel column: Switches are off and disabled; coach clicks "Connect" inline CTA → routes to integration setup page for that provider

### Autonomous Mode Selection

1. Coach opens Settings → Autonomous Mode
2. RadioGroup shows current setting (Manual / Mode B / Mode A) — default Manual
3. Selecting Manual or Mode B → immediate save, success toast: "Saved — Manual mode active." / "Saved — Auto-send after 24h active."
4. Selecting Mode A → AutonomousModeAConfirmModal opens; RadioGroup highlights Mode A but does NOT commit yet
5. In modal: coach types `send without review` (exact match). Confirm button enables.
6. Coach clicks "Enable autonomous send" (destructive fill) → modal closes, save commits, warning banner appears above RadioGroup
7. Coach clicks Cancel or dismisses modal → RadioGroup reverts to previous selection silently

### Tokenized Review Link Flow

1. Coach receives email/Slack/WhatsApp/SMS with a review link
2. Clicks link → loads `/review/{token}` (or `/r/{token}` → 302 → `/review/{token}` for SMS)
3. Server verifies JWT (HS256, `JWT_REVIEW_SECRET`), checks expiry (7d), checks nonce against `consumed_tokens`
4. **If valid + unconsumed:** renders ReviewPage with DraftCard in glass surface, Approve/Edit/Hold buttons enabled
5. **If consumed:** renders Already-actioned state with "Open dashboard" CTA
6. **If expired:** renders Expired state
7. **If invalid signature:** redirect to `/r/invalid` static page
8. Coach taps Approve → POST to `/api/review/{token}/action` → server consumes nonce, acquires advisory lock, runs approval → page transitions to Already-actioned glass card with "Approved" heading and "Open dashboard" CTA
9. Read-only viewing (page load without action) does NOT consume the nonce — coach can re-open the link to read the draft until they take action

### Slack Block Kit Interaction

1. Coach receives Slack DM with Block Kit message
2. Taps "Approve & send" (primary green) → Slack POSTs to `/api/webhooks/slack/interactivity` with signed payload
3. Server: verifies `X-Slack-Signature` (HMAC-SHA256, 5-minute timestamp tolerance) → acquires Postgres advisory lock keyed on `hashtextextended(draft_id::text, 0)` → atomic status transition → Gmail send → calls Slack `chat.update` to replace the message with the "approved" confirmation
4. If draft already approved by another channel (lock acquired but status no longer pending): Slack message updates to "Already approved" + responds with ephemeral message "This draft was approved from another channel."
5. Tapping "Edit" opens the Slack modal pre-filled with current draft body. On submit: server validates, saves edits, runs approve+send in one atomic step, updates the original DM to "Edited and approved"
6. Tapping "Hold" triggers Slack-native confirm dialog → on confirm: status → `held`, DM updates to "Held"

### 24h Follow-up + 48h HOLD Cascade

1. Draft created → `notification/draft_ready` Inngest event fires immediately, parallel fan-out to all enabled channels
2. If `drafts.status` still `pending` 24h after `notifications.sent_at`: Inngest scheduled step fires `notification/draft_followup_cta` event, increments `drafts.followup_count` to 1
3. Same dispatcher, different copy template (subject and SMS body use "Reminder:" prefix)
4. If still `pending` 24h later (48h total) with `followup_count = 1`: Inngest cascade step transitions `status → held`, sets `drafts.held_at = now()`, increments `followup_count` to 2 (sentinel)
5. Held drafts surface in the Held tab automatically via Realtime; no notification fires for the auto-hold transition (coach already received 2 notifications)

## Accessibility

All elements meet WCAG 2.1 AA.

| Requirement | Implementation |
|-------------|---------------|
| Focus management | All interactive elements reachable via Tab. Held DraftCard auto-focuses first card on tab change. Tokenized review page auto-focuses DraftCard's first action. |

| Tokenized review page (logged out) | Page renders fully accessible — no auth modal, no skeleton trap. Errors render as semantic alerts. |
